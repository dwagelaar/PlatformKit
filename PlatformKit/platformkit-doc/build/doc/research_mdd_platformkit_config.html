Profiling and validation

Each configuration model that is expressed in a platform-aware configuration language can be checked by PlatformKit to see for which platforms it is valid.

When editing a product configuration model in the tree-based EMF editor, one can add new child elements to the model. For the instant messenger configuration language, the options are listed in Fig. 1. EMF already takes care of multiplicity constraints given in the meta-model by greying out the invalid options. PlatformKit can further limit the available options by removing the options that are invalid for a chosen platform:

  • Right-click the root element of the configuration model and select “PlatformKit → Profile Against Concrete Platform...”, as shown in Fig. 2.
  • Select a platform description from the list of built-in platforms and click “OK”, as shown in Fig. 3. Alternatively, select “Platform specification from file” after which you can provide a platform specification “.owl” file from the workspace. Example platform specifications can be found here.
Normally, the built-in Pellet OWL DL reasoner is used to perform the above steps. It is also possible to use an external OWL DL reasoner via the DIG interface. This is done by selecting “Window → Preferences...” from the menu and changing the settings in the “PlatformKit” category.

As a result, the amount of configuration options has been reduced to valid options only. Fig. 4 shows the valid options for configuring an instant messenger for a JDK 1.1 PC platform. Any existing configuration choices can be validated against the chosen platform:

  • Right-click the root element of the configuration model and select “Validate”.

If there are any invalid elements in the configuration model, they will be reported, as shown in Fig. 5.
